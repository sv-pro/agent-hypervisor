The Problem

Agents inherit every risk
of the tools they touch.

What you deploy
tools:

What you implicitly allow

arbitrary_exec(shell, subprocess)
Risk Surface

Three things break before you notice.

Compliance
Agents can compose actions your policy engine never anticipated. No audit trail. No approval gate. GDPR, SOC 2, HIPAA exposure surfaces at the intersection of tools, not within any single tool.
Predictability
The model decides at runtime. The same task description produces different action sequences across runs. You cannot certify behavior you cannot enumerate.
Blast Radius
A single injected instruction can re-task a multi-step agent. Without structural limits on what the agent can express, no behavioral guardrail holds under adversarial input.
Auditability
"The agent did it" is not a control narrative. Regulators, insurers, and boards need deterministic accounts of what was possible, not just what happened.
The Standard Approach

The action space stays open.

The Architecture

Capability rendering eliminates
the gap.

Capability Rendering
At design time, define exactly what the agent can express.
At runtime, construct only those expressions.

The agent does not pick from available tools. It operates within a projected ontology — a purpose-built action vocabulary for its current task context.

email-assistant manifest (rendered world)
capabilities:
reply_to_sender(draft: str) # NOT send_email(to: any, body: any)
summarize_thread(thread_id: str) # NOT read_file(path: any)
flag_for_review(reason: str) # NOT modify_calendar(any)
─────────────────────────────────────────────
# "send to external address" cannot be proposed — it is not in the ontology

Integration

It fits into what you already have.

IAM / RBAC
Manifests map to roles. Capability sets become the permission boundary, not individual API scopes. Existing RBAC infrastructure gains structural enforcement.
SIEM / Audit Logs
Provenance traces emit structured events — capability used, taint level, verdict, context. Feed directly into existing SIEM pipelines without instrumentation overhead.
Policy Engines
Governance rules operate on manifests, not tool calls. 12 manifests are easier to certify than 847 rules over unbounded combinations. Formal coverage becomes possible.
Existing Tooling
No model replacement. No new runtime. The rendering layer sits between your tools and the model's action space — a seam, not a fork.
Architecture

Four layers. One control surface.

L0
Execution Physics
Sandbox isolation, process boundary, network egress controls. Unchanged from current practice.
L1
Base Ontology
Full capability inventory — every tool, every parameter. Defined at integration time by platform engineers.
L2
Dynamic Ontology Projection ← capability rendering lives here
Task-scoped subset of L1. The model only perceives what the manifest allows. Dangerous actions are not visible.
L3
Execution Governance
Policy engine + provenance firewall. Verdict (allow / deny / ask) per action. Append-only trace.
