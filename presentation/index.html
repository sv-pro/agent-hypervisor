The Inversion
The dominant paradigm asks
"Can agent X do action Y?"
Behavioral question. Assumes the action exists in the world. The answer is probabilistic. It is bypassable.
The right question is
"Does action Y exist in agent X's world?"
Ontological question. If the action has no existence, there is nothing to permit or deny. The answer is structural. It is not bypassable.

Behavioral restriction: the world is dangerous, the agent is constrained.     Ontological construction: the world is safe, the agent is free.

The Architecture

Four Layers of World Design

The agent lives inside this structure. It cannot see the boundaries. The boundaries are the security.

L3
Execution Governance
What actions may execute — deterministic policy evaluation, reversibility classification, budget enforcement, approval gates. No LLM on this path.
L2
Dynamic Projection · Rendering
What actions the actor can propose now — context-specific capability subset rendered at runtime. The action space is narrowed to what is safe in this context. This is the central insight.
L1
Base Ontology
What actions exist in this system at all — the typed capability vocabulary, the world schema, the semantic event types. Authored at design-time.
L0
Execution Physics
What is physically impossible — enforced by construction, not policy. The hard floor. Below this layer, nothing exists.

"An action outside the ontology cannot be proposed."

Layer 2 — The Central Insight

The Rendering Engine

A policy engine asks:
"Is this action allowed?"

A rendering engine constructs:
"Here is the action space that exists for this actor, in this context, right now."
This is not a permission check.
It is a reality-rendering engine.
The agent does not navigate a restricted world.
It navigates a purpose-built world.
Rendered world · context: git workflow

The narrowing happens before the agent reasons. The agent cannot even form an intent for an action that does not exist.
Architecture Principle

Stochastic at Design-time.
Deterministic at Runtime.

Design-time
LLM + human — collaborative ontology authoring
Manifest — world schema, capability vocabulary
Compiler — validates, type-checks, produces artifacts
Red-team — stochastic adversarial simulation
Output — deterministic, static policy artifacts
Runtime
No LLM — on the critical execution path
Rendering — loads context, projects capability set
Governance — deterministic policy evaluation
Same input → same decision. Always.
Unit-testable — every physics law is a test case
Intelligence is in the design. Enforcement is in the structure. These two concerns must not share the same execution path.

The Stakes

These are architectural failures.

Every current class of agentic attack is a consequence of the same gap: the agent lives in raw reality. There is no governing layer between the agent and the world.

Prompt Injection
Untrusted input reaches the agent as undifferentiated text. The agent cannot distinguish instruction from data. A typed perception layer removes the ambiguity by construction.
Tool Misuse
General-purpose tools (Bash, HTTP, file I/O) give the agent a surface area far larger than any task requires. A rendered capability world removes unused surface at design-time.
Memory Poisoning
Agent memory has no provenance, no taint markers, no trust levels. Information written by attackers and trusted principals is stored identically. Semantic isolation solves this structurally.
MCP / Supply Chain Risk
Agents that consume raw tool output from untrusted servers inherit whatever intent those servers embed. A governed input boundary intercepts this before the agent sees it.
These are not separate problems with separate solutions. They are one architectural gap with one architectural answer.
The Missing Layer
