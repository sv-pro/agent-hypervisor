Objection 01
"Isn't this just guardrails with extra steps?"
Answer

Guardrails observe behavior and block it after the model has proposed it. Capability rendering means the model never proposes the dangerous action — it is not in the ontology the model perceives.

This is the difference between a locked door and a room that does not have the door. One can be jailbroken. The other cannot — the action is not expressible.

# Guardrail model:
model proposes send_email(to="attacker")
classifier catches it — sometimes
─────────────────────────────
# Capability rendering:
model has no send_email(to: any) in its world
cannot propose what does not exist
Objection 02
"What if the model finds a way to bypass the rendering layer?"
Answer

"Bypass" requires the model to call a capability that is not in its rendered ontology. The enforcement is not in the model — it is in the execution layer. The model's output is validated against the manifest before any tool call executes.

The model cannot "bypass" by clever prompting, because the constraint is not a prompt. It is enforced at the tool-call dispatch boundary — outside the model's reach.

Structural constraints are not prompt-level. Jailbreaks operate on text. This operates on dispatch.
Objection 03
"Why not just remove the dangerous tools from the agent?"
Answer

Because the same tool is safe in one context and dangerous in another. send_email is a legitimate capability for a customer support agent and an exfiltration vector for a document summarizer.

Removing tools eliminates utility globally. Capability rendering makes a tool safe by restricting how it can be invoked — constraining recipients, content scope, attachment access — per workflow.

same tool, different manifests
# customer-support manifest
send_reply(to=thread.sender, body=str) ← safe
# document-summarizer manifest
— send_email not in ontology ← absent
Objection 04
"Where exactly is the control point? Who enforces it?"
Answer

The control point is the tool-call dispatcher — the component that receives the model's intended action and dispatches it to actual execution.

Before dispatch, the hypervisor validates three things: (1) the capability is in the active manifest, (2) the arguments satisfy the manifest's parameter constraints, (3) the provenance of inputs does not violate the policy. Only then does execution proceed.

Model output (tool call intent)
↓ validate against manifest
Hypervisor dispatch (control point)
↓ allow / deny / ask
Tool execution (if allowed)
Objection 05
"What if the model selects the wrong manifest / misidentifies the task?"
Answer

Manifest selection is not left to the model in production deployments. The orchestrator assigns the manifest based on the declared workflow — deterministically. The model operates within the assigned ontology; it does not choose it.

In research or open-ended modes, the model can request a manifest. That request itself goes through policy — a manifest that includes request_manifest_change has a governance rule governing it.

The model is a tenant of the rendered world. It does not administer it.
Objection 06
"This seems over-engineered. We just need better prompts."
Answer

Better prompts are stochastic. For any given prompt, there exists an adversarial input that causes the model to behave differently. This is not a hypothesis — it has been demonstrated across every frontier model.

Structural constraints are not stochastic. An action that is not in the manifest cannot be proposed regardless of prompt crafting. You cannot prompt your way out of a structural boundary.

Prompt-based safety

Brittle under adversarial inputs.
No coverage proof.
Behavior changes with model updates.

Structural safety

Invariant to prompt crafting.
Coverage is provable (closed set).
Behavior survives model updates.
